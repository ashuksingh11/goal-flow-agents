Here's where I'd push, roughly in priority order.

**1. Your cloud/device boundary is doing two jobs at once.** You've drawn one line, but it's actually two independent axes — reasoning-vs-execution _and_ cloud-vs-edge — and you've collapsed them into "planning = cloud, doing = device." The meal optimizer breaks that, because it needs both heavy reasoning (argues cloud) _and_ the freshest possible local state, i.e. what's literally in the fridge right now (argues edge). So the real question isn't "which tier plans," it's "which way does the data flow" — does the cloud pull inventory up to plan, or does the device pull a plan/model down and optimize locally? And that's driven by what you're actually optimizing for, which I can't yet tell: privacy, latency, offline resilience, or inference cost? **Which one is it?** The answer reshapes the whole boundary. My instinct: cloud orchestrates + talks + remembers; the device is the _sole_ authority on local state and the only thing that touches actuators; planning is a negotiation, not owned by one side.

**2. Here I disagree outright — don't put hard constraints and soft preferences in the same memory.** "Daughter dislikes mushrooms" and any medical/allergen line are being pulled from the same fuzzy store. That's fine for dislikes and dangerous for anything safety-critical: semantic retrieval can silently miss, and a missed preference is a bad dinner while a missed allergen is a health incident. Split a small, authoritative, _always-injected_ safety/constraints profile (allergies, medical, hard dietary/religious lines) from the large fuzzy preference + episodic memory, and have the safety gate check plans against that hard profile **deterministically** — not "hope the model remembered." This is the one place I'd refuse to rely on retrieval at all.

**3. The week-long lifetime is the hard part and currently the thinnest.** A seven-day operation has to survive device reboots, cloud session timeouts, and a user who ignores an approval for two days. So: where does task state actually live, and who is the source of truth — cloud, device, or both with reconciliation? You need durable, resumable workflow state, idempotent steps (a retried "add to shopping list" mustn't add twice), timeouts with escalation on stale approvals, and — critically — a reconciliation step, because a plan made Monday is stale Wednesday once someone's eaten the spinach. **What's your intended source of truth and durability model?**

**4. Disagreeing again — approving everything will kill the product.** A week of "should I add X? should I do Y?" produces one of two failures: users rubber-stamp (so approval is theater and your safety guarantee is gone) or they disengage. Blanket human-in-the-loop is the wrong default. Tier actions by reversibility × cost × risk: auto-execute reversible low-stakes ones with notify-and-undo (a prep reminder), notify-only for medium, and require explicit approval only for irreversible/costly/allergen-adjacent actions (spending money, placing an order). Make autonomy configurable and let it grow with trust. "Ask forgiveness with easy undo" beats "ask permission" for anything reversible — and it cleanly separates what the safety gate must _block_ from what it merely _logs_.

**5. Your flow is linear; the real system is event-driven.** The football match is treated as a one-off but it's the general case. You need a loop reacting to state changes (calendar added, inventory depleted, meal skipped), a _materiality_ policy (is this change big enough to replan?), and a rule for whether a change patches only the affected days or re-opens the whole plan. Patch-don't-rebuild is better for stability and trust — and when a change collides with an already-approved action, decide now what wins.

**#2 (constraint split) — my suggestion.** Two stores, and the important move is _where the hard block lives_. Keep a small, structured, authoritative profile — allergens, medical, hard religious/dietary lines — that is **never** retrieved semantically; it's a fixed object. Keep everything else (dislikes, patterns, episodic history) in the large fuzzy memory that biases planning. The key: the hard block travels _inside the task contract_ as a first-class field, so the device doesn't re-derive it. The device's safety gate then does a literal set-membership check — does any dish contain something on the deny-list → block. For the POC this is almost free: a tiny JSON with `hard_constraints` vs `soft_preferences`, and the gate is a few lines of plain code, not an LLM "please avoid." That split — the LLM _plans_, code _checks_ — is itself a pattern worth showing, and it seeds the final design cleanly.

**#3 (durability) — POC-scoped.** Device holds task state **in memory**, since it's already the authority on local state — no persistence, no reconciliation, no crash recovery for the demo. The one thing I'd not fake: give the task a real ID and an explicit status lifecycle (`created → planning → awaiting_approval → executing → adapting → done`) as a serializable object. That satisfies your "keep it in memory" instruction literally, and shapes it so the final design just swaps the store underneath. It also happens to be exactly the create/assign/check-status spine you mentioned wanting — that lifecycle _is_ the task-management layer.

**What each tier does in the POC**

Cloud agent — _understand, remember, talk._ Owns the dialogue and takes the fuzzy goal; runs one round of ambiguity resolution ("when you say healthy… when you say next week…"); holds the hard profile + soft memory and retrieves the relevant bits; decomposes into the task contract — the _what and why_ (objectives, hard/soft constraints, time window), **not** the _how_; dispatches the contract, then relays the device's plan, proposals, and questions to the user and collects approvals; presents the final plan and owns the adaptation conversation.

Device agent — _execute, local truth._ Receives the contract; creates the task, assigns the ID, drives the status lifecycle; pre-checks access to the (mocked) sources; discovers capabilities from a mocked registry; gathers local state — inventory + calendar; produces the concrete per-day plan (this is where the _how_ lives, because only the device knows what's in the fridge); runs the deterministic safety gate; emits the plan plus side-effects as **proposals, not actions** (shopping adds, prep tasks); on approval, executes against mocked actuators and updates status; and on a scripted calendar change, proposes an adjustment back up.

The linchpin is the **task contract** — that interface _is_ the orchestration pattern. **Keep it two genuinely separate agents, not one pretending to be two.** The strongest temptation in a demo is to collapse cloud and device into a single reasoning loop with a stylistic label swap.
